Version 0.12 ships two improvements:

First, the background warmup system got stronger: app services can now start warming up early when you call `.preassemble()`, usually on the main service of your app, instead of waiting for a later `.assemble(...)`.

Second, the service lifecycle API got smaller: the `lazy: boolean` flag is gone, as adding it to the framework caused unnecessary complexity. A simple pattern can be used to make a factory lazy or eager, no need for a library flag.

## Change 1: Adding .preassemble()

In v0.11, writing `service(name).app({...})` validated the dependency graph and built a reusable assembly blueprint, but the factory itself stayed cold. Nothing ran until an explicit `.assemble({...})` call somewhere else in your program.

In the current v0.12 design, declaration still stays cheap. If you want earlier warmup, you opt into it by calling `.preassemble()`, usually on your main service. That triggers an initial assembly with whatever supplies are already known and kicks off a background warmup pass over the whole reachable graph. For request-independent services, that means their factories can finish building at startup. For request-dependent services, the preassembly attempt just memoizes a "missing dependency" error on the unresolved part of the graph and waits for real request data to arrive. The error is only thrown if the service's value is accessed before the needed request data has been provided.

This keeps startup compute under your control. If you want request-independent services ready before the first request, call `.preassemble()`. If you do not, typectx avoids doing that extra work up front.

```typescript
const $main = service("main")
    .app({
        services: [$router],
        factory: ({ router }) => router
    })
    .preassemble()
```

In practice, that one call is usually enough: preassembling the main service lets typectx warm up the whole request-independent part of the graph and cache it across requests.

## Change 2: The Lifecycle API Gets Simpler

Separately, v0.12 simplified the lifecycle API (lazy and init). Two things changed:

- `lazy: boolean` is removed.
- `init` is renamed to `warmup`.

### `lazy: boolean` is gone

In v0.11, every app service accepted a `lazy` flag:

```typescript
// v0.11
const $reportGenerator = service("reportGenerator").app({
    factory: () => new ReportGenerator(),
    lazy: true
})

const $app = service("app").app({
    services: [$reportGenerator],
    factory: (deps) => (action: "view" | "report") => {
        if (action === "report") {
            // Factory only runs here, because deps.reportGenerator is a getter
            return deps.reportGenerator.generate()
        }
    }
})
```

The flag had two real problems:

1. **It conflated "when the factory runs" with "when the product is used."** A single boolean decided both, so you couldn't, for example, lazily build a service in tests and eagerly build it in production.
2. **Destructuring was a trap.** Because `deps` was an object of getters, writing `factory: ({ reportGenerator }) => ...` on a lazy service immediately triggered the factory, which was not necessarily what you wanted.

v0.12 deletes the flag. Laziness is now expressed by **what your factory returns**.

### The three factory patterns

With `lazy` gone, here are the three shapes your factories can take. All of them are plain factories from typectx's point of view—the difference is purely in what your code returns.

**1. Eager factory (the default).** Return your product directly. typectx builds it when the service is assembled, and also during `.preassemble()` if you opt into startup warmup, then memoizes the result.

```typescript
const $eagerService = service("eagerService").app({
    services: [$db],
    factory: ({ db }) => buildExpensiveService(db)
})
```

**2. Lazy factory.** Return a memoized function, and put the real work inside that function. The expensive work only runs the first time a consumer calls it.

```typescript
import { once, service } from "typectx"

const $lazyService = service("lazyService").app({
    services: [$db],
    factory: ({ db }) =>
        once(() => {
            return buildExpensiveService(db)
        })
})
```

No more destructuring gotcha: the factory body is cheap, the expensive work is inside `once(...)`, and consumers just call `deps.lazyService()` when—and only when—they actually need the product.

**3. Warmed-up factory.** Return a memoized function exactly like the lazy pattern, then use the `warmup` callback to decide whether to call it eagerly:

```typescript
const lazy = false
const $warmedService = service("warmedService").app({
    services: [$db],
    factory: ({ db }) => once(() => buildExpensiveService(db)),
    warmup: (lazyExpensiveService) => {
        if (lazy) return
        lazyExpensiveService()
    }
})
```

This pattern allows to easily flip between eager and lazy without refactoring the codebase: this service returns a function no matter if it is run eagerly or lazily.

### `init` → `warmup`

The old `init` callback had a fine job: run right after `factory` returns, receive the product and deps, do some setup work. In practice, that's exactly what "warming up" means—populate caches, kick off memoized functions, log readiness—so the name has been changed to match the intent.

```typescript
// Before (v0.11)
const $profile = service("profile").app({
    services: [$currentUser],
    factory: () => memo((id) => db.profiles.get(id)),
    init: (getProfile, { currentUser }) => {
        getProfile(currentUser)
    }
})

// After (v0.12)
const $profile = service("profile").app({
    services: [$currentUser],
    factory: () => memo((id) => db.profiles.get(id)),
    warmup: (getProfile, { currentUser }) => {
        getProfile(currentUser)
    }
})
```

The function signature is identical. Only the key name changed.

## Migration Guide

### 1. Delete `lazy: true` and return a memoized function instead

```typescript
// Before
const $expensive = service("expensive").app({
    services: [$db],
    factory: ({ db }) => buildExpensiveService(db),
    lazy: true
})

// Consumers used property access
deps.expensive
```

```typescript
// After
import { once, service } from "typectx"

const $expensive = service("expensive").app({
    services: [$db],
    factory: ({ db }) => once(() => buildExpensiveService(db))
})

// Consumers call the memoized function
deps.expensive()
```

You can also use the warmed-up pattern if you are unsure if eager or lazy loading is the best option for the performance of your service.

```typescript
const $expensive = service("expensive").app({
    services: [$db],
    factory: ({ db }) => once(() => buildExpensiveService(db)),
    warmup: (build) => {
        build()
    }
})
```

### 2. Rename `init:` to `warmup:`

No signature change, just a key rename:

```typescript
// Before
service("x").app({ factory: ..., init: (value, deps) => { ... } })

// After
service("x").app({ factory: ..., warmup: (value, deps) => { ... } })
```

### 3. Add `.preassemble()` if you want startup warmup

If you want request-independent services warmed up before the first request, add `.preassemble()` to your main service:

```typescript
const $main = service("main")
    .app({
        services: [$user],
        factory: ({ user }) => user
    })
    .preassemble()
```

If you do not call `.preassemble()`, declaration stays cheap and warmup happens later when you call `.assemble()`.

## Why These Changes?

1. **Earlier work when you want it, less compute when you do not.** - Let preassemble() optimize your app for you.
2. **Simpler mental model for lifecycle.** Eager, lazy, and warmed-up are now patterns you write in ordinary JavaScript—not flags inside a framework option bag.
3. **No destructuring trap.** The lazy pattern uses a function call, which is both obvious to read and impossible to accidentally trigger through destructuring.

typectx is inching closer to v1. Each release has been chipping away at the "one way to do it" surface, and v0.12 does that both under the hood, with explicit opt-in preassembly, and at the public API, with the shrunken lifecycle options.
